What is `this` all about?

By Vanessa Correia

Posted August 17, 2020 in JavaScript Fundamentals

Reading time: 4 minutes.

this is a JavaScript keyword that refers to the object it belongs to. However, it has different values depending on where it is used. Let’s break this down so as to fully undertand what is is, where and why it is used and why the scope of it plays an important rule.

First things first, lets list all the possible values this may assume:

Alone, this refers to the global object.

If we were to simply type and run console.log(this) in our terminal, we would see a very long and scary object being print into our console.

That is the global object, which in JavaScript, it is always defined.

Similarly, if we were to open our console in the browser and type and run console.log(this) as well, we would see that our code comes up as being part of the Windown object.

In a function, this refers to the global object.

It was previously mentioned that this is a JavaScript keyword that refers to the object it belongs to. So what happens if we use this in a regular function?

const playlist = {
	title: "Roadtrip Songs",
	play() {
	console.log(this);
	}
}

function playSongs() {
    console.log(this);
}

playSongs();

Notice that the playSongs function is not declared within the playlist object.

Therefore, this now logs the global object (which will be window in the browser and the global object in Node).

In a function, in strict mode, this is undefined.

In the previous example, we witnessed default binding. Something to take into consideration is that if we replicated the previous example in strict mode, this would be undefined, because default binding is not allowed in strict mode.

In an event, this refers to the element that received the event.

This one is maybe easier to visualize with an example:

 <button onclick="this.style.display='none'">
  Click to Remove Me!
</button>

As we can see, the onclick event is attached to the button, therefore this will be tied to that specific event, in that specific button.

In a method, this refers to the owner object.

A method is basically a function within an object, acting as a property of said object. Lets illustrate this with an example:

const playlist = {
	title: "Roadtrip Songs",
	play() {
	console.log(this);
	}
}

playlist.play();

Since play() is a method in the playlist object, this is referencing the object itself:

{ title: 'Roadtrip Songs', play: [Function: play] }

We can therefore conclude that if we use this from withing an object method, it will point to the “owner” of the method, which in this case is playlist.

This would remain true even if we added further methods to playlist:

const playlist = {
	title: "Roadtrip Songs",
	play() {
	console.log(this);
	}
}

playlist.stop = function() {
    console.log(this);
}

playlist.stop();

{ title: 'Roadtrip Songs',
  play: [Function: play],
  stop: [Function] }

This would not be the case if we were using a constructor function:

const playlist = {
	title: "Roadtrip Songs",
	play() {
	console.log(this);
	}
}

function Playlist(title) {
    this.title = title
    console.log(this);
}

const p = new Playlist("Running Songs")

Since we are making use of the new keyword to create a new instance of Playlist, this now references a new object:

Playlist { title: 'Running Songs' }

Another thing we should consider is the behaviour of this within callback functions.

const playlist = {
	title: "Roadtrip Songs",
	genres: ["pop", "rock", "indie"],
	displayGenres() {
	this.genres.forEach(function(genre) {
	console.log(this.title, genre);
	})
	}
}

playlist.displayGenres()

The code above results in the following:

undefined 'pop'
undefined 'rock'
undefined 'indie'

Why? Because this is still referencing the global object. Although we are inside the playlist object, the function were we are attempting to make use of this is a regular function, hence the undefined.

In order to solve this, what we can do is make use of the second parameter (thisArg) of the forEach:

const playlist = {
	title: "Roadtrip Songs",
	genres: ["pop", "rock", "indie"],
	displayGenres() {
	this.genres.forEach(function(genre) {
	console.log(this.title, genre);
	}, this)
	}
}

playlist.displayGenres()

By doing so, we are able to achieve the intended result:

Roadtrip Songs pop
Roadtrip Songs rock
Roadtrip Songs indie

We would ran into a similar problem if we attempted to use an arrow function, since arrow fuctions always bind to the global object.

Methods like call() and apply() can refer this to any object.

Although call() and apply() are methods, they are predefined JavaScript methods.

This means that both of them can be used to call an object method with another object as argument:

 let playlist1 = {
  playlistTitle: function() {
    return this.firstSong + " " + this.lastSong;
  }
}

let playlist2 = {
  firstSong:"Run to You",
  lastSong: "7 Rings",
}
playlist1.playlistTitle.call(playlist2);  // Will return "Run to You 7 Rings"

As we can see, this refers to playlist2, even if it is a method of playlist1.

In conclusion

Although it seems an easy enough concept to grasp, this can become a slippery slope if we don’t take into account how the way it is used may play a role in its value and, ultimately, in the accuracy and correctness of our code.

Main concepts to keep in mind are the fact that in can be used in strict mode and that will affect what this will bind too. The behaviour of this is also quite diferent depending on the type of function we are using - object method, pre-defined method, regular functions, callback functions and even arrow functions.
